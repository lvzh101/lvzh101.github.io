Windows与Linux本地Nginx代理配置指南

这篇文章整理了一个本机通过 Nginx 接管 api.openai.com 请求的完整方案，覆盖 Windows 和 Linux 两种环境，并总结了常见报错、排查方法和长期稳定运行建议。

这套方案的核心思路是：

  • 通过 hosts 把 api.openai.com 解析到本机
  • 由本机 Nginx 接收 443 请求
  • 再由 Nginx 转发到上游服务，例如 https://right.codes/codex/

先说明定位：

  • 这套方案更适合本机调试、学习、内网实验
  • 不推荐作为公网正式生产方案
  • 如果追求长期稳定，最佳实践依然是：自己的域名 + 正规 DNS + 正规证书 + Linux Nginx

一、整体原理

以 Windows 为例，链路是这样的：

  1. 程序或浏览器访问 https://api.openai.com/v1/...
  2. Windows 的 hosts 把 api.openai.com 指向 127.0.0.1
  3. 本机 Nginx 监听 443
  4. Nginx 把请求转发到上游，例如 https://right.codes/codex/v1/...

Linux 也是同样的原理，只不过：

  • Windows 改的是 C:\Windows\System32\drivers\etc\hosts
  • Linux 改的是 /etc/hosts

8. Windows 开机自启与长期运行建议

如果只是双击 nginx.exe，那：

  • 关机后进程会消失
  • 重启系统后需要重新启动

因此更推荐把 Nginx 注册为 Windows 服务。

常见做法：

  • 使用 NSSM 把 nginx.exe 注册成服务
  • 或使用“任务计划程序”设置开机启动

如果希望长期稳定，推荐：

  • hosts 只保留 IPv4 映射
  • 关闭浏览器安全 DNS
  • 避免代理软件接管 api.openai.com
  • 将 nginx.exe 注册成系统服务

6. 浏览器不通，但 curl 正常

这种现象非常常见。

常见原因：

  • 浏览器 DNS 缓存
  • 浏览器连接池缓存
  • 代理软件接管了请求
  • 安全 DNS 绕过了本机 hosts
  • 浏览器扩展改写了网络行为

解决方法：

  • 执行 ipconfig /flushdns
  • 清 Edge 的 DNS 和 socket 缓存
  • 关闭浏览器安全 DNS
  • 关闭代理/VPN/代理扩展
  • 优先访问 /healthz 判断是不是走到了本机 Nginx

7. 浏览器提示 ERR_CONNECTION_CLOSED

通常说明：

  • 本机 Nginx 没成功监听 443
  • 或证书握手被浏览器直接拒绝
  • 或浏览器走了和 curl 不同的网络链路

先确认：

如果 curl 成功而浏览器失败，优先排查浏览器侧缓存、代理和 DNS。

五、代理软件场景下的建议

如果电脑上运行了代理软件，比如：

  • Clash
  • Clash Verge
  • Mihomo
  • v2rayN
  • 其他 TUN / 系统代理软件

最稳的做法不是让代理去代理 api.openai.com，而是：

  • 让代理软件彻底不要碰这个域名

建议原则：

  • api.openai.com 走 DIRECT
  • 127.0.0.1 走 DIRECT
  • localhost 走 DIRECT
  • 禁止代理软件用 Fake-IP 或远程 DNS 接管这个域名

七、最后的建议

这套“hosts + 本机 Nginx + 反向代理”的方案，优点是：

  • 上手快
  • 本机验证方便
  • 不需要额外买域名

但它的缺点也很明显：

  • 容易受到浏览器缓存、代理软件、系统网络栈影响
  • 长期维护成本高
  • 不适合正式公网长期使用

因此建议这样选择：

  • 只是自己本机调试： 这套方案可以用
  • 要长期稳定： 最好迁移到 Linux，并使用自己的域名
  • 要正式上线： 自己域名 + 正规 DNS + 正规证书 + Linux Nginx 才是最佳实践

如果只是临时测试，这篇文章里的配置已经足够；如果以后要做成长期方案，建议尽早切换到正规部署方式。
