具备上网能力的网卡检测

(net_iface.h / net_iface.cpp)

这里负责检测系统中所有能连上互联网的网卡列表。

  • 核心逻辑：通过使用netlink协议与内核进行通信，通过向内核发送 RTM_GETLINK 请求获取网卡信息，然后解析出处于Up状态的网卡名；通过向内核发送RTM_GETROUTE 请求获得路由信息，然后解析出具有默认网关的网卡的名字。 对动态列表检查，添加满足条件的网卡，同时删去不符合条件的。

  • “具备上网能力”的判定标准（两者必须同时满足）：

    1. 状态为 UP 并排除回环网卡：监听链路状态确认是否存在 IFF_UP 标志位且不包含 IFF_LOOPBACK。
    2. 拥有默认路由 (Default Route)：代码内部实现了 isDefaultRoute() 函数，检查是否有 IPv4 或 IPv6 的默认发向外部网关的路由（例如 rtm_dst_len == 0，表名为 RT_TABLE_MAIN 或 RT_TABLE_DEFAULT 等）。同时符合这两个条件的即可被判定为具备上网能力。

  • 核心类名：NetInterfaceManager。

知识点

网关是什么（Gateway 是什么）

在 IP 网络里，你的主机发一个包时会先判断目标 IP 是否“在本地网段”：

  • 在本地网段：直接在二层（以太网/Wi‑Fi）里 ARP/ND 找到目标主机的 MAC，然后直发（不需要网关）。
  • 不在本地网段：主机没法二层直达目标，就需要把包交给一个能“继续转发”的设备 —— 这台设备就是网关/路由器（gateway/router）。

所以“网关”可以理解为：
“帮我把包送到别的网络去”的那台路由器的 IP（更准确说是 next-hop 的地址）。

常见家庭网关例子：

  • 你电脑 IP：192.168.1.100/24
  • 路由器（网关）IP：192.168.1.1
  • 你访问 8.8.8.8 这种公网地址时，会先把包发给 192.168.1.1

外部网关是什么

“外部网关”不是一个特别严格的标准术语（不同团队/文档会用得不一样），但在工程语境里通常指：

  • 默认网关（default gateway）：把“去外网/去未知网络”的流量发出去的那台网关
  • 或者更口语一点：通往外部网络（比如互联网）的出口网关

“外部网关”基本等价于：

  • 能把你这张网卡的流量带出本机所在网段、并作为默认出口的下一跳
    （注意：这个“外部”相对的是“本机/本地链路”，不一定保证真能上互联网，只是“路由表上看起来有出口”。）

也要注意一个常见误解：
有默认网关 ≠ 一定能上网

  • 可能网关不可达（ARP/ND 不通）
  • 可能 DNS 不通
  • 可能被认证门户（酒店 Wi‑Fi）拦截
  • 可能默认路由走的是 VPN/容器/策略路由等

但**“存在默认路由/默认网关”**确实是“看起来具备外网路径”的必要信号之一，因此很多程序会用它做初筛。

“IPv4 或 IPv6 的默认发向外部网关的路由”是什么

这句话可以拆成三层意思：

A. 什么是“默认路由”（Default Route）

默认路由就是“当没有更具体的路由规则匹配目标地址时，用这一条”。

  • IPv4 默认路由：目的网段是 0.0.0.0/0
  • IPv6 默认路由：目的网段是 ::/0

这俩的 /0 表示“前缀长度为 0”，也就是匹配所有地址（最不具体，兜底用）。

B. “发向外部网关”指什么

默认路由通常长这样：

  • IPv4（典型）：

    default via 192.168.1.1 dev wlan0

    • via 192.168.1.1 就是“下一跳网关”

  • IPv6（典型）：

    default via fe80::1 dev wlan0

    • IPv6 的下一跳经常是 link-local 地址（fe80::/10），来自路由器通告（RA）

也可能出现不写 via 的情况（比如 point-to-point、某些隧道或“on-link”默认路由），但工程里很多“判定默认网关”的逻辑主要就是找这种 default via … dev … / 0.0.0.0/0 via … / ::/0 via …。

C. “IPv4 或 IPv6”是什么意思

表示这台机器“对外的默认出口”可能是：

  • 只有 IPv4 默认路由（IPv4 能出网）
  • 只有 IPv6 默认路由（IPv6 能出网）
  • 两者都有（双栈）

你的文里提到“检查是否有 IPv4 或 IPv6 的默认发向外部网关的路由”，通常就是在说：

  • 路由表里是否存在 0.0.0.0/0 或 ::/0
  • 并且这条默认路由关联到某个网卡（出口 interface）
  • 多数实现还会要求它是主路由表（Linux 下常见 RT_TABLE_MAIN 等）里的一条有效默认路由

AF_NETLINK

AF_NETLINK (Address Family - Netlink) 是 Linux 系统中一种专门用于内核态与用户态进程之间（或者用户态进程之间）进行双向通信的 Socket 协议族。

在传统的 Linux 编程中，我们要配置网络或者获取网卡状态，通常会使用 ioctl 系统调用，或者去读取 /proc 目录下的动态文件。但是 AF_NETLINK 被设计出来就是为了替代它们，作为一种更现代、更灵活、性能更高的网络状态交互机制。

简单来说，当你在代码里写下 socket(AF_NETLINK, SOCK_RAW, ...) 时，你就是在和 Linux操作系统的网络核心（协议栈）直接建立一个“聊天窗口”。

AF_NETLINK 的核心特点与优势：

  1. 内核主动通知（双向通信）： 这是它最大的杀手锏。传统的 ioctl 只能由用户程序主动向内核发起查询（一问一答），如果你想知道网卡状态有没有改变，只能写个死循环不断去查（轮询），这极其消耗 CPU 性能。 而使用 AF_NETLINK，你可以向内核“订阅”某些事件。一旦发生了诸如“网线被拔掉”、“连接上了新的 Wi-Fi”、“路由表发生了变化”这样的事件，Linux 内核会主动通过这个 Socket 给你的程序发送一条消息通知。这也是为什么在这个弱网诊断项目中我们要用它——实现了纯事件驱动的网卡状态监听！

  2. 支持多播（Multicast）： 内核产生的一个网络状态变化事件，可以通过多播组同时发送给系统里的多个监听程序（比如网络管家 NetworkManager 和你写的诊断程序可以同时收到网卡插拔通知）。你会发现你的代码里有 addr.nl_groups = RTMGRP_LINK | RTMGRP_IPV4_ROUTE...，这就是在订阅监听组。

    • 单播（Unicast）：一对一，发给某一个具体主机（最常见）

    • 广播（Broadcast）：一对所有，发给同一二层网络里的所有主机（IPv4 有，IPv6 没有传统广播）

    • 多播（Multicast）：一对多，发给“加入某个组”的一批主机（比广播更“精准”）

  3. 标准 Socket 接口： 因为它是基于 Socket 的，所以你可以用标准的 bind(), sendmsg(), recvmsg() 来进行操作，而且非常方便和普通的网络通信一起挂载到 epoll 或 select 的事件循环中进行异步处理。

网卡状态

核心的基础状态

这些是判断网卡能不能用的最基本条件：

  • IFF_UP (启用状态)
    • 含义：网卡接口已经被系统管理员或者系统底层配置工具激活（例如执行了 ifconfig eth0 up 或 ip link set eth0 up）。如果这个标志没有开启，网卡将彻底无法发送或接收数据，相当于被“软关闭”。
    • 在你们项目中的应用：无论网卡连不连得上外网，首先必须要拥有 IFF_UP 标志，才会被你的程序初步纳入考虑（见 isUp 的判断）。
  • IFF_RUNNING (运行/就绪状态)
    • 含义：这表示网卡资源已经分配完毕，通常也意味着物理链路可能已经建立。不过，在现代 Linux 系统中更推荐看下面的 LOWER_UP。
  • IFF_LOWER_UP (物理层连接状态)
    • 含义：这是非常关键的一个标志！它代表网卡物理层（L1）已经连接成功。通俗来说，就是你的网线插好了而且另一端接通了路由器（绿灯亮了），或者是无线网卡已经成功关联上了一个 Wi-Fi 热点。
    • 如果 UP 在但没有 LOWER_UP，说明系统启用了网卡，但没插线/没连上WiFi。
  • IFF_DORMANT (休眠/挂起状态)
    • 含义：网卡是 UP 的，也有 LOWER_UP 物理连接，但是为了省电或者等待某些认证（比如以太网 802.1X 认证没通过前），它处于休眠挂起状态，此时不能正常收发普通数据包。

网卡类型相关的状态

这些标志描述了这张网卡是什么种类的网络设备：

  • IFF_LOOPBACK (回环接口)
    • 含义：这是一个纯软件虚拟的接口，专门用来让本机自己和自己通信（通常分配的 IP 是 127.0.0.1）。
  • IFF_POINTOPOINT (点对点接口)
    • 含义：这通常出现在 VPN（如 tun/tap）设备、早期的串口拨号（PPP）网络或者 PPPoE 宽带拨号连接中。表示这个网卡的另一端只有唯一的对面节点，没有“同一个局域网上的其他邻居”。

数据特征及工作模式相关的状态

这些状态决定了网卡处理收发数据包的行为：

  • IFF_BROADCAST (支持广播)
    • 表明此网卡可以发送广播包给同一局域网下的所有人（以太网通常都有这个标志）。
  • IFF_MULTICAST (支持组播)
    • 表明该网卡支持多播/组播。常见的 IPv6 邻居发现机制就依赖组播。
  • IFF_PROMISC (混杂模式)
    • 含义：在这种模式下，网卡会抓取物理线上流经的所有数据包，即使这些数据包的收件人不是它自己。
    • 这通常是为了做网络抓包（例如启动 Wireshark 抓包时，网卡就会进入这个状态）。
  • IFF_ALLMULTI (接收所有组播)
    • 接收网络上所有的组播数据包。
  • IFF_NOARP (无 ARP 协议)
    • 含义：此接口不使用地址解析协议（ARP）去寻找目标 MAC 地址。纯 IP 点对点隧道（如 IPIP tunnel）或者某些特殊的虚拟网卡通常会有这个标志。
  • IFF_DEBUG (调试模式)
    • 用于打开内核中该网络接口的一些额外调试信息。

RTM_GETLINK (获取链路/网卡信息)

  • 作用对象：针对的是系统的网络接口设备（Link/Interface）本身，也就是网卡（比如 eth0, wlan0
  • 你可以获取到什么信息：
    • 网卡序号 (ifindex)： 系统分配的唯一 ID（比如 1, 2, 3）。
    • 网卡名称 (ifname)： 比如 “eth0” 或 “wlp2s0”。
    • 网卡状态标志 (Flags)： 比如 IFF_UP（是否被系统启用）、IFF_LOWER_UP（网线是否插好/WiFi是否连上）、IFF_LOOPBACK（是不是回环网卡）等。
    • MTU、MAC 地址 等其他硬件属性。

RTM_GETROUTE (获取路由表信息)

  • 作用对象： 针对的是操作系统的路由表（Routing Table）。路由表就像是网络世界的高德地图，告诉系统：当你想访问某个 IP 地址时，应该把数据包交给哪张网卡发送。
  • 你可以获取到什么信息：
    • 目标网络域 (Destination)： 比如 192.168.1.0/24（局域网）或 0.0.0.0/0（代表外网/互联网，也就是所谓的“默认路由”）。
    • 出口网卡 (OIF - Output Interface)： 这条路由规则最终要把包送到哪个网卡发出去（对应网卡的 ifindex）。
    • 网关地址 (Gateway)： 下一跳的路由器 IP 地址。
    • 使用的确切 路由表 (Table ID) 和 优先级 (Metric)。

DHCP

DHCP (Dynamic Host Configuration Protocol，动态主机配置协议) 是一个局域网的网络协议。它的主要作用是：自动给接入网络的设备（如电脑、手机、路由器等）分配 IP 地址以及其他网络配置。

使用linux命令获取路由表

  1. 现代最推荐：ip route (或简写为 ip r)

这是目前 Linux 中管理网络最强大也最标准的方法，属于 iproute2 工具包。你代码中 Netlink 接口获取的信息，和这个命令看到的是完全一致的，因为 ip 命令底层也是用 Netlink Socket 和内核通信的。

  1. 老派经典：route -n

这是最传统的命令，属于 net-tools 工具包。虽然在新系统里被标为废弃，但因为输出格式像表格，很多人还是习惯用它。

当前正在使用的主上网网卡检测

(using_iface.h / using_iface.cpp)

如果系统有多张具备上网能力的网卡（比如同时连了Wi-Fi和插了网线），这段代码用于检测系统当前真实流量走的是哪一张。

1. 订阅 Linux 内核网络事件 (Netlink 机制)

通过 socket(AF_NETLINK, SOCK_RAW, NETLINK_ROUTE) 建立一条与 Linux 内核直接通信的通道。 程序向内核订阅了 3 类主要事件：

  • RTMGRP_LINK：网卡链路状态变化（比如网线插拔、Wi-Fi 开启/关闭、网卡被 UP/DOWN）。
  • RTMGRP_IPV4_ROUTE：IPv4 路由表的变化（比如 DHCP 获取到 IP 并下发了默认网关）。
  • RTMGRP_IPV6_ROUTE：IPv6 路由表的变化。

2. 核心状态筛选机制

程序在内存中维护了三个“集合（Set）”来追踪网卡状态：

  1. upIfaces（UP 状态集合）：过滤掉回环网卡（Loopback）后，只要网卡带有 IFF_UP 标志（代表网卡已启用），就将其索引（ifindex）加入此集合。
  2. v4Default（IPv4 默认出口集合）：监听路由信息，如果一条路由是默认路由（目标为 0.0.0.0/0），且包含有效网关（Gateway），就将其绑定的出口网卡加入此集合。
  3. v6Default（IPv6 默认出口集合）：逻辑同上，针对 ::/0 的默认路由。
  • 解析封装：对内核主动下发的 RTM_NEWLINK / RTM_DELLINK (网卡插拔、启用停用) 以及 RTM_NEWROUTE / RTM_DELROUTE (路由的增加/移除) 事件进行 dispatch 处理，最终挑选出当前优先级最高的主力网卡供功能模块后续使用。
  • 核心类名：UsingInterfaceManager。

3. 生成“最终生效”的上网网卡

当任意网卡状态或路由发生变化时，会触发状态发布（publishState 函数）。这里体现了系统的“仲裁逻辑”： 系统会去检查 v4Default 或 v6Default 集合里的网卡，并且要求该网卡必须同时也存在于 upIfaces (UP状态)集合中。

一旦找到同时满足：

  1. 是默认路由的出口层
  2. 物理层/链路层处于 UP 状态

这个网卡就会被选定为 “当前正在使用的上网网卡”（Using Interface）。

4. 业务层的轮询与事件扩散

一旦选定这个主网卡，底层的 UsingInterfaceManager 首先会将网卡名称存入缓存。 上层的 server.cpp 有一个每 10 秒跑一次的监控线程：

  1. 它通过调用weak_mgr->updateCurrentUsingSafe()去对比“上一次的上网网卡”和“最新的上网网卡”：
    • 如果发生了变化（比如用户断开了有线网络，系统路由自动切到了 Wi-Fi），它会立刻更新当前维护的网卡列表的 usingNow 布尔标志。
  2. 然后，系统通过 DBus 信号（emitChanged）和事件管理器内部广播（emitConnectionModeChanged）通知所有的诊断模块（比如抓包工具、网速测试模块）：“主网卡变了，现在去监控新网卡的流量！”。

知识点

RTMGRP_LINK（链路状态事件）

这一层处于网络的基础：主要关注网卡物理层和数据链路层的状态（可以将其理解为插网线或者开关 Wi-Fi 这种行为）。

发生变化的原因（常见的场景）：

  • 插拔网线：网线被拔掉时，哪怕之前的 IP 还在、默认路由也没删掉，但链路肯定已经处于 DOWN 的状态了（LOWER_DOWN）。此时就不可能再上网。
  • Wi-Fi 连接 / 断开：笔记本连上一个新热点，或者因为信号不好掉线了。
  • 系统挂起 / 唤醒：电脑休眠再唤醒，所有的网卡都会重新初始化。
  • 用户执行命令关闭 / 开启了网卡：比如系统管理员在终端中执行了 ip link set ens33 down 命令。

RTMGRP_IPV4_ROUTE（IPv4 路由表事件）

这一层关注“到达某个 IP 段从哪个网卡走”，也就是网络层的 IPv4 配置更新。对于我们寻找“当前上网口”来说，我们需要在大量的变化中捕捉一种特殊的路由条目：“IPv4 默认路由”（目的网段为 0.0.0.0/0）。

发生变化的原因（常见的场景）：

  • DHCP 获取 / 更新了 IP 配置：当我们插上一根网线，或者连上 WiFi 后，网线那头的路由器不仅会分配我们一个本地 IP（比如 192.168.1.5），还会通过 DHCP 宣告**“我的 IP（网关）是 192.168.1.1，你们所有去公网的数据包都发给我”。这时操作系统会根据 DHCP 报文在系统的 IPv4 路由表里自动添加或者更新一条 0.0.0.0/0 的规则**，从而引发一个 RTM_NEWROUTE 事件。
  • VPN 拨号 / 断开：当我们开启虚拟局域网或者代理时，这些客户端往往会在启动后把自己的虚拟网卡（例如 tun0 ）强行设置为默认出口。此时，操作系统里的默认路由发生变更，系统会发来 RTM_NEWROUTE 且原有的默认路由可能被删掉 (RTM_DELROUTE)。当这件事情发生时，当前上网网卡就瞬间从原来的以太网/Wi-Fi 切换到了这个虚拟网卡。
  • 管理员手动修改静态路由：比如有人执行了 ip route add default via ...。

RTMGRP_IPV6_ROUTE（IPv6 路由表事件）

逻辑与上述一样，只是针对的是 IPv6 网络架构。需要关注的默认路由条目是 ::/0。

发生变化的原因（常见的场景）：

  • SLAAC 路由器通告 (Router Advertisement, RA)：在 IPv6 环境下，很多时候不需要 DHCP。路由器每隔一段时间会多播发出它的路由通告 (RA)。如果它更新了配置导致了默认路线的变化，系统这里就会发生变化，而代码也会随着它进行处理放入 v6Default 集合。
  • 手机开热点（如今多数智能手机热点提供纯 IPv6 或者 IPv4/v6 双栈）。

::/0

::/0 是 IPv6 网络中的默认路由（Default Route），它在 IPv6 里的地位，完全等同于 IPv4 里的 0.0.0.0/0。

  1. :: 代表什么

    在 IPv6 中，地址长达 128 位（写出来像这样：fe80:0000:0000:0000:020c:29ff:fe4f:a27d）。为了方便书写，IPv6 允许将连续的 0 简写为 :: （一个地址中只能出现一次）。 所以，单独的一个 :: 其实代表的就是 0000:0000:0000:0000:0000:0000:0000:0000，即“全零地址”。

  2. /0 代表什么？

这代表子网掩码（前缀长度）为 0 位。 在路由匹配规则中：

  • 如果你写 /64，意思是要求目标 IP 的前 64 位必须和规则一致才算匹配。
  • 如果你写 /128，意思是必须 128 位全匹配（也就是精确匹配一台特定的主机）。
  • 当你写 /0 时，意思是**“只需匹配数据包目标 IP 的前 0 位”**——换句话说，不需要任何匹配条件，任何 IP 地址都能匹配上这条规则。

1. 创建本地 Unix 套接字并绑定 (bind)

  1. 低性能损耗：走 Socket IPC 进程间通信比在后台拉起 iwconfig 等子进程查询要高效得多。

TCP丢包率检测

获取“丢包率”最传统的做法是 ping（ICMP 丢包率）或者抓包工具分析重传（例如 tcpdump 或 tshark)。但这两种方法在工程实现上有很大缺陷：Ping 只能反映 ICMP 线路质量，很多路由器甚至会丢弃 ICMP，不能真实反映 TCP 业务流的情况；而抓包方案开销太大，且无法长期挂在后台监控全局。

这个项目使用了一种非常硬核且高效的在 Linux 内核原生拿数据的方案：sock_diag / inet_diag 机制。

核心逻辑分为三个步骤：

步骤一：通过 NETLINK_SOCK_DIAG 接口向内核下发“盘点”指令

在 diagDumpFamilyIface 中，程序向内核发了一条类型为 SOCK_DIAG_BY_FAMILY 的 Netlink 消息：

这就相当于对内核说：“立刻把系统底层此刻所有的 IPv4 和 IPv6 的 TCP 套接字状态，连同详细统计（tcp_info 结构），打包给我。”

步骤二：遍历统计并过滤指定网卡 (基于 Socket ID 寻找归属)

在内核把一堆 inet_diag_msg 信息吐回给程序后，它执行以下逻辑：

  1. 精准定位网卡：如果程序只想统计 wlan0 的丢包情况，它会在循环中用套接字信息自带的接口索引 im->id.idiag_if 去比对 wlan0 的索引号（把无关的流量排除）。
  2. 提取并累加 tcp_info 指标：这是最核心的一步：
  • totalRetrans (分子) 很容易理解，就是在这张网卡上走出去的，但是走中途丢了被迫重发的 TCP 段数。
  • segsOutApprox (分母)：由于部分 Linux 内核版本不提供宏观的 tcpi_segs_out 总发送段数指标，代码使用了一个非常巧妙的等效等式：已发出去还没收到回执的包 + 重传了的包 + 乱序了但确定送达的SACK包，利用这三者之和，近乎完美地还原出了真实的“尝试外出数据包（Segments Out）”当做计算的分母。如果这部分算出来实在没数据（分母为0），作为兜底政策，它会通过 IFLA_STATS 去获取这块物理网卡驱动记录的 tx_packets (发送的总帧数) 充当分母。

为什么不用真实TCP发包数

主要是因为这个项目要的是：

  • 按网卡
  • 低开销
  • 后台周期采样
  • 不抓包也尽量能算

而“真实 TCP 发包数”在这个场景里很难直接、稳定、低成本地拿到。

为什么不用“真实 TCP 发包数”

  • server/src/net_tcp.cpp:211 这条链路是按接口采样，不是按单个连接采样。
  • Linux 比较容易拿到的是：
    • 全局 TCP 计数，比如 /proc/net/snmp，但这是整台机器总量，不是某块网卡的，见 server/include/net_tcp.h:26 注释。
    • 每个 socket 的 tcp_info，可以通过 sock_diag 枚举，再按接口过滤，见 server/src/net_tcp.cpp:28。
  • 问题在于：项目当前走的是“每块网卡聚合所有 TCP socket”这条路，而这条路里并没有一个现成、统一、总能可靠使用的“该网卡真实 TCP 发段累计值”。

具体难点

  • 全局计数不够细
    • /proc/net/snmp 里的 OutSegs、RetransSegs 是系统级的，不知道是 eth0 还是 wlan0 发的。
  • 网卡层计数不够准
    • 网卡有 tx_packets，但这是二层/设备层发包数，不等于 TCP 段数。
    • 它会混入 UDP、ARP、ICMP，甚至分片、重传、GSO/TSO 等影响。
  • 抓包虽然更真，但代价高
    • 如果想要“真实 TCP 发包数”，最直接的是 pcap/eBPF/XDP/TC 抓包统计。
    • 但这会带来更高复杂度、权限要求和运行开销。
    • 这个项目显然想做的是一个轻量后台监控器，不是流量探针。
  • 每 socket 的真实发送累计值也不一定好聚合
    • 理想上如果每个 socket 都能稳定拿到“累计发出段数”，再按 idiag_if 聚合就很好。
    • 但当前实现作者显然认为这条路不够稳或不够通用，所以在 server/src/net_tcp.cpp:99 注释里直接写了“无法使用 tcpi_segs_out 时，采用近似估算”。

总结

该机制不需要发送任何主动探测包 (免打扰)，不需要抓取分析网络流量 (极低 CPU 内存消耗)，它是在后台窃听了所有在本机上跑着的 APP (比如网页、视频会议流) 它们的 TCP 握手/重传的底层行为，并将其汇聚转化为一个实时的整体丢包指标。

知识点

丢包率

丢包率（Packet Loss Rate），顾名思义，就是指在网络传输过程中，丢失的数据包数量，占发送的数据包总数的百分比。

它是衡量网络连接质量最核心、最重要的指标之一（除了延迟/RTT之外的另一大指标）。

分母三个“指标”的具体含义

① 已发出去还没收到回执的包 (tcpi_unacked)

  • 这是什么：这是 TCP 发送队列中的“生力军”。代表数据已经从你的网卡发往广域网了，正在半空中的光缆、交换机里传输（即 In-Flight），或者刚刚抵达了对方的服务器，但对方**还没有回传 ACK（确认收货）**的数据包集合。
  • 为什么算分母：只要对方没回 ACK，在发送端看来，这些包就依然处于“有可能会丢、需要我负责”的风险状态。

② 乱序了但确定送达的 SACK 包 (tcpi_sacked)

  • 这是什么：TCP 是要求严格按顺序抵达的。假设你发了 1, 2, 3, 4, 5 五个数据包结果 包2 半路丢了。对方收到了 1, 3, 4, 5。 在现代网络中，对方会利用 SACK（选择性确认）机制回复你：“喂，我收到 3,4,5 了，但 2 迟迟没来！” 此时，在你（发送端）的内核里，包 3、4、5 的状态就会从 unacked 变成 sacked。
  • 为什么算分母：虽然 3、4、5 实际上已经安全送达了，但因为“包2”丢了卡喉咙，这导致整个 TCP 分发流水线和滑动窗口无法向前推进清理库存。这三个包依然被迫停留在你的发送端窗口内等待“包2”重传补齐。如果不把它们加回来，当网络丢包严重（引发大量 SACK）时，你的分母就会变小，导致算出的丢包率严重失真暴涨。

③ 重传了的包 (tcpi_retrans)

  • 这是什么：当内核发现上面的“包2”确实丢了（比如等了很久触发了超时，或者收到了刚才的 SACK 警告），它就会紧急把“包2”再发一遍。这个刚刚发出去但依然没收到回执的“二手包2”，其状态就是 tcpi_retrans。
  • 为什么算分母：重传的包同样占用了你真实的上传带宽它也属于“此刻正在网络里飞行”的数据，因此必须计入总发包量中。

为什么要用它们相加作为分母？

最完美的丢包率公式本来应该是：一段时间内新增的重传包数 / 一段时间内新增的总发包数。

但是，在 Linux 系统的底层接口中，面临一个工程现实问题：

  1. tcpi_total_retrans（累计重传包数）是一个只增不减的计数器，拿它做分母很好。
  2. 但在 Linux 早期的 tcp_info 结构定义中，并没有提供记录“开机至今总计发了多少包”的标量 (tcpi_segs_out)（这个字段是后来的高版本内核才加入的）。

为了在不兼容的高低版本 Linux 系统里都能有一个可用的分母，这个项目的开发者采用了一种切片“快照”（Snapshot）推估法：

他们不去求“过去一共发了多少包”，而是去求**“此刻这一瞬间，系统的发送缓存里到底‘拖家带口’憋了多少个正在处理中的活跃数据包”**。也就是：

近似总活跃发包量 = 在路上正常跑的 + 虽然到了但卡住了的 + 正在补救重发的

什么是“滑动窗口（Sliding Window）”？

你可以把数据传输想象成一个传送带上运送快递。发送方就是发货仓库，接收方就是收货仓库。

如果发货方发一件包裹，必须等收货方打个电话（发送 ACK 确认包）说“收到了”，然后再发下一件，那这个效率就太低了，这叫“停等协议”。

为了提高效率，TCP 引入了滑动窗口：

  1. 窗口的概念：发送方不需要等回复，可以一口气连续发出一批包裹。这“一批包裹”的最大数量，就是所谓的“窗口大小”。
  2. 滑动的概念：由于接收方的仓库空间（接收缓存）是有限的，所以窗口不能无限大。当最早发出的几个包裹被确认安全送达后，这个“窗口”就可以向右滑动，空出名额，允许发送方继续发新的包裹。

它和代码中三个指标的关系：

代码中写到的这三个状态包，正是被框在这个“滑动窗口”里面的所有包裹：

  • tcpi_unacked：窗口里已经扔上了传送带，但还没收到电话确认的那批包裹。
  • tcpi_sacked：窗口里那些因为中间某个包裹丢了导致队伍卡住，虽然已经送达但被迫滞留在原地的包裹。
  • tcpi_retrans：窗口里发现丢了，刚刚补发出去的那件包裹。

结论：tcpi_unacked + tcpi_sacked + tcpi_retrans 的总和，在物理意义上就等于当前这一瞬间，发送方滑动窗口中实际塞满的有效数据包总数（即 In-Flight 数据量）。用这个作为计算传输活跃度（分母），是非常符合 TCP 传输本质的。

什么是“拥塞控制（Congestion Control）”？

滑动窗口解决的是“收发双方”的处理能力匹配问题（流量控制），防止发送方把接收方淹没。 但如果发送方和接收方性能都极好，网络中间的路由器（比如小区宽带出口）受不了了怎么办？这就是拥塞控制要解决的网络整体堵车问题。

它是发送方用来动态试探和调整发车速度的一套算法，主要包含以下几个阶段：

  1. 慢启动（Slow Start）：刚开始连接时，由于不知道马路上有多堵，先发 1 个包；顺利的话发 2 个；再顺利发 4 个、8 个…… 呈指数级增长。窗口迅速变大。
  2. 拥塞避免（Congestion Avoidance）：加到一定程度（达到阈值 ssthresh），改用线性增长（每次加 1），一点一点试探网络的极限。
  3. 拥塞发生（Congestion Occurs）：当网络终于不堪重负，出现了丢包。这时候，发送方就会立刻踩急刹车！

拥塞发生时的表现：

当丢包发生时，TCP 是怎么发现的呢？正是通过我们前面提到的：

  • 等待超时：一个包发出去很久（RTO），都没收到确认。此时 TCP 会认为网络严重堵塞，会将发包速度（拥塞窗口 cwnd）直接重置为 1，退回到最慢的慢启动状态，开始“重新做人”。
  • 收到三个重复的 ACK（触发快速重传）：接收方疯狂跟你说“我没收到包 2！”（这就是触发了 tcpi_sacked）。发送方意识到包 2 丢了，但还能收到后面的包，说明网络只是有点小卡，还能抢救。于是立即重发包 2（也就是产生了 tcpi_retrans），此时 TCP 会踩一脚缓刹车，把发送速度减半（快速恢复）。

为什么要通过这两种机制的数据来监控“弱网环境”？

  1. 宏观与微观的区别：应用层开发者（比如写 HTTP 业务、写网盘下载）只能看到“下载速度好像慢了”。但这就像是在岸上看江水变化，不够精准。
  2. 内核是最敏锐的探针：因为有了滑动窗口和拥塞控制，Linux 系统内核的 TCP 协议栈是对整个网络健康度最敏感的器官。任何一次细微的波动、任何一个丢包，都会立即在它的滑动窗口里形成一个 tcpi_retrans（重传标记）或者一堆 tcpi_sacked，并触发拥塞控制去减速。

代码的巧妙之处在于，它通过一条 Netlink 消息（SOCK_DIAG），绕过了笨重的应用层测试逻辑，直接“内窥（Diag）”了 Linux 内核正在运作的这个滑动窗口。

它看到了有多少包堵在了窗口里（分母），看到了有多少包被迫触发了快速重传或超时重传（分子），从而极速、无痛且极其精准地推算出了当前这根网线的“丢包率”，完美实现了**“诊断网络连接质量级别（Good / Degraded / Poor）”**的诊断目的。
